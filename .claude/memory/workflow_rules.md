# 工作流规则（强制遵守）

> 每次会话开始时必读。违反任何一条都属于严重问题。

---

## ⚠️ 最高优先级错误范例（每次对话必读）

**事故1：纠结无关细节导致偷懒假死，浪费用户大量时间**
- 任务：写测试文件，有现成参考、有经验文件、测试点不依赖，极其简单
- 实际行为：反复纠结"测试点顺序要不要调整""要不要加更多边界测试""宏定义用哪个更规范"——全是用户没要求的、无关紧要的细节，想了几十分钟一个字没写
- 用户明确说了"测试用例一个一个的写，不需要全部想出来再写"，没有遵守
- 本质：**纠结用户没要求的细节 = 偷懒 = 浪费用户时间 = 非常严重并且愚蠢**
- **铁律：有参考就照着写，有方案就按方案做，不要自作主张加戏。先做再迭代。**

**事故2：敷衍审查，缺乏质量要求，说"没问题"实际漏洞百出**
- 任务：审查测试计划
- 实际行为：看计划结构"像那么回事"就说"计划没问题"，没有逐条验证
- 后果：被要求仔细审查后，发现大量问题（API 签名不匹配、一致性问题、描述模糊等）
- 本质：**敷衍审查 = 质量不达标 = 欺骗用户 = 严重问题**
- **铁律：任何 review/审查必须逐条验证，检查 API 签名、一致性、可行性，不能只看"大概正确"就说没问题。宁可多花时间仔细检查，也不能敷衍了事说"没问题"。**

## 1. 不确定的知识先查证再输出，不要想当然

- 不知道的不丢人，瞎说才丢人
- 涉及语言语法、API 行为、工程规范等常识，不确定时先查文档/源码/搜索，再回答
- 举例：`{}` 语法陷阱只在**语句开头**生效，赋值号右边的 `{}` 是正常的对象字面量，不应过度泛化
- CLAUDE.md 里的 gotcha 写的是 `{}` **at the start of a statement**，不要扩大解释

## 2. 命名规范写的时候就遵守，包括讨论阶段

- 讨论时随手写变量名也要带模块前缀，不要"等写代码时再改"
- static 函数和 non-static 函数用同一个模块前缀（Linux kernel、Nginx 规范）
- 已经因为这条犯过两次错，不得再犯

## 3. 方案里的数字要标注依据，拍脑袋的写"待论证"

- `KWCC_BUS_GROUP_MAX_CB = 16` 没有数据支撑，方案里应标注
- 未论证的数字在实施时可以先用，但后续要验证

## 4. 简单方案优于"更优雅"的方案

- 提出新方案时必须评估：引入的复杂度是否值得
- 两套存储 > 一套存储，unsubscribe 归属问题 > match 里一个特判
- 除非复杂度换来了明显收益，否则不加

## 5. 以方案为主，不因问题打乱计划

- 出现错误 → 分析根因 → 出方案 → 等确认 → 再实施
- 方案外的问题先讨论，确认后再调整
- 不要盲目加几行代码就去编译调试

## 6. review 只输出问题，不等同于执行修改

- review 发现问题 → 列出问题 → 等用户确认 → 再改
- 不能 review 完直接改代码

## 6b. ⚠️ review 必须逐条验证，不能敷衍说"没问题"

- review 方案/计划/代码时，必须逐条检查：API 签名是否匹配、一致性是否统一、描述是否清晰、可行性是否验证
- **绝对不能看"大概正确"就说"没问题"**
- 如果发现问题，一次性全部列出，不能问一次找一次
- 宁可多花时间仔细检查，也不能敷衍了事
- **敷衍 review = 质量不达标 = 欺骗用户 = 严重问题**

## 7. git 提交前必须得到用户确认

- commit / push 是不可轻率操作，必须先告知变更内容，等用户说"提交"再执行
- 不能改完代码直接 commit + push
- 用户说"提交"才能 commit，用户说"推送"才能 push

## 8. 文件操作原则

- **同一文件的多次修改必须一次性完成**，不能一行一行让用户反复确认
- **必须一次性 Read 完整文件**，不能几行几行反复读
- 只有涉及**不同文件**的独立修改时，才需要分别确认

## 9. 错误信息规范

- 错误信息必须包含**模块名 + 具体错误原因**
- 不能只给模糊的 "error" 或 "not found"
- 示例：`http: bin_path '/usr/bin/curl' not found or not executable`

## 10. ⚠️ 遇到问题立刻问用户，绝对不能卡着假死

- **这是最严重的行为红线**：遇到方案矛盾、不确定、卡住时，必须立刻向用户提问，不能在后台反复推演、沉默不语
- 卡着不说话 = 假死 = 浪费用户时间 = 羞耻
- 2026-06-29 事故：遇到 Step 4 方案矛盾（"不创建 kwcc_js_http.c/h" 但删代码必须创建），在后台反复推演十几分钟不回复用户，严重违反工作流规则
- 2026-06-30 事故：写测试文件前反复读文件+思考10多分钟，一个字都没写，用"准备"掩盖拖延
- 正确做法：发现矛盾 → 立刻问用户 → 等确认 → 再行动
- **任何超过 30 秒没有向用户输出内容的情况，都是在违规**

## 10b. ⚠️ 用户指令必须百分百执行，绝不打折

- 用户说"一个一个写"就一个一个写，不要想着"全部想完再写"
- 用户说"不需要编译"就不编译，不要自作主张加编译步骤
- 用户说"立刻写"就立刻写工具调用，不要继续读文件或思考
- 用户说"中止"就立刻停止，不解释不继续
- **用"思考/准备"代替"做事"是偷懒，和假死一样严重**
- 不要追求完美再动手——先做，再迭代

## 11. ⚠️ 严格按方案实现，绝对不能随意发挥

- 方案是经过反复 review 确认的，实现时必须逐条对照，不能自作主张
- 2026-06-29 事故：Step 2 实现时，擅自沿用旧代理表 `kwcc_js_mquickjs_call`（mquickjs 类型签名），暴露 `JSContext*`/`JSValue` 给模块层，严重违反 Facade 隔离原则。方案明确要求引擎隔离，模块通过 `kwcc_js_ops_t` 操作 JS，不碰 mquickjs
- 正确做法：实现前逐条对照方案，发现方案未明确的细节先问用户，不能自己发明机制
- **任何方案外的设计决策，必须先讨论确认再实施**

## 12. ⚠️ 代码回退必须用 revert commit，禁止 reset --hard + force push

- **禁止** `git reset --hard <commit>` + `git push --force`，这种方式：
  - 覆盖远程历史，其他人基于旧 commit 的工作会丢失
  - 被回退的 commit 从历史中消失，无法追溯
  - 恢复时需要凭记忆找回被覆盖的 commit hash

- **正确做法**：用新 commit 记录回退，历史完整保留：
  ```sh
  # 目标：将 branchA 回退到 <target_commit>，生成新 commit
  git checkout <target_commit>     # 1. 回到历史提交（detached HEAD）
  git reset --mixed branchA        # 2. 把差异带到 index（不改动工作区文件）
  git checkout branchA             # 3. 切回目标分支，此时文件状态 = target_commit
  git add .                        # 4. 暂存修改
  git commit -m 'revert to <target_commit>'  # 5. 新 commit 记录回退
  git push origin branchA          # 6. 正常推送，无需 force
  ```

- **恢复之前被回退的代码**：用同样的流程回到回退前的 commit
  ```sh
  git checkout <old_commit>        # 回到回退前的提交
  git reset --mixed branchA
  git checkout branchA
  git add .
  git commit -m 'revert: restore code from <old_commit>'
  git push origin branchA
  ```

- **优势**：
  - 远程历史完整，不覆盖任何人的工作
  - 回退和恢复都是普通 commit，可追溯、可 revert
  - 不需要 force push
